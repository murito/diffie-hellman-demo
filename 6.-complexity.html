Algorithm Complexity & Quantum

O(1) Constant Time

The execution time is independent of the input size. Accessing an array element by index or basic arithmetic.

Scroll to increase complexity
O(log n) Logarithmic Time

Execution time increases very slowly. Classic example: Binary Search. Highly efficient even for massive datasets.

O(n) Linear Time

Performance is directly proportional to the input size. Iterating through a list or finding the max value.

O(n log n) Linearithmic Time

Standard for efficient sorting algorithms like Mergesort and Quicksort. Good for large scale processing.

O(n²) Quadratic Time

Nested loops. Performance degrades quickly. Bubble sort or naive matrix addition.

O(nᵏ) Polynomial Time

The boundary of "tractable" problems. While $O(n)$ and $O(n^2)$ are polynomials, higher degrees like $O(n^{10})$ become extremely slow but still mathematically distinct from exponential growth.

2^O(nε) Subexponential Time

Faster than exponential but slower than polynomial. This is the complexity of the best classical algorithms for Discrete Logarithms and RSA.

O(2ⁿ) Exponential Time

Doubles with each new input. Brute-forcing passwords or solving the Towers of Hanoi.

O(n!) Factorial Time

The "Hard Limit". Calculating all permutations of a set. Traveling Salesman Problem (Brute Force).

CYBERSECURITY CHALLENGE

The Discrete Logarithm Problem

This math problem powers modern encryption (Diffie-Hellman). We assume it is nearly impossible to solve for large numbers... until Quantum computing arrives.
