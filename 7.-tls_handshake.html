TLS & Diffie-Hellman

The Role of DH in TLS

In modern web security (TLS 1.3), the Diffie-Hellman algorithm is used to perform a "Key Exchange" before any sensitive data is sent. It allows the browser and the server to establish a shared secret over a public connection.

This process provides Forward Secrecy: even if the server's long-term private key is stolen later, past conversations remain encrypted and secure.

The Legacy RSA Key Exchange

Before DH became standard, RSA was used for both authentication and key transport:

1. ClientHelloSupported ciphers and random numbers sent.
2. ServerHello & CertificateServer sends its RSA Public Key inside a Digital Certificate.
3. Certificate ValidationClient verifies the certificate using pre-installed CAs.
4. Premaster Secret ExchangeClient generates a random secret, encrypts it with the Server's RSA Key, and sends it.
5. Decryption & Session KeyServer uses its Private Key to decrypt the secret. Both sides now share the key.
Trust vs. Mathematics
RSA / Authentication

Relies on Previous Contact. The system only works because your browser comes pre-installed with "Trusted Root CAs". Without this prior setup, you couldn't trust the RSA keys.

Diffie-Hellman / Key Exchange

Relies on Pure Math. Two strangers can establish a shared secret with no previous contact or pre-installed keys. The math creates the secret on-the-fly across the wire.

Modern TLS combines RSA (to prove who the server is) and DH (to create the secure key).

Why DH instead of RSA?

Older versions of TLS (1.2 and below) often used RSA for key exchange. However, this had a major flaw: if an attacker recorded the traffic and later stole the server's private key, they could decrypt all past traffic.

Perfect Forward Secrecy (PFS) Using Ephemeral Diffie-Hellman (DHE) means the keys are generated per-session and discarded immediately after use. Stealing the server's certificate provides no help in decrypting recorded sessions.
